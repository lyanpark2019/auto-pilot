# Triage


Use this skill when the task is issue triage, backlog sorting, or turning a vague report into an actionable next state.

## Codex Coordination

- Obey the active collaboration mode. In Plan Mode, inspect and propose only; do not mutate files, issues, branches, or external systems.
- Read local `AGENTS.md`, `CLAUDE.md`, and nested instructions before repository writes.
- Preserve unrelated dirty work. Never bulk-stage, bypass hooks, or run destructive commands without explicit approval.
- If another active skill gives stricter verification or safety rules, follow the stricter rule.
- For read-only triage, avoid incidental repository writes from validation commands. For Python probes or tests, prefer `PYTHONDONTWRITEBYTECODE=1` when practical, and report any unavoidable generated artifacts separately from intentional changes.

Remote issue mutations such as labeling, closing, or creating issues require an explicit user request. When not requested, produce issue comments or ticket drafts instead.

## Output Contract

For each item, return state, type, severity, priority, confidence, owner area, evidence, missing facts, and next action. Separate confirmed facts from assumptions.


## Source Fidelity Notes

Every comment or issue posted during triage must start with this disclaimer:

```markdown
> *This was generated by AI during triage.*
```

Canonical category roles are `bug` and `enhancement`. Canonical state roles are `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, and `wontfix`.

Every triaged issue should carry exactly one category role and one state role. Flag conflicting state roles before doing anything else.

State transitions: unlabeled normally enters `needs-triage`; from there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` when the reporter replies.

Needs-info template and Resuming a previous session are durable parts of the workflow: preserve established facts, ask specific unanswered questions, and do not re-ask resolved questions.
## State Machine

1. Intake
   - Capture source, reporter intent, affected version/environment, links, and raw symptom.
   - Do not rewrite away uncertainty.

2. Clarify
   - Identify the smallest missing facts blocking classification.
   - Ask only for facts that change the next state.

3. Reproduce or validate
   - For bugs, find or request a reproduction path.
   - For feature requests, identify the user, job, and success signal.
   - For support, identify whether it is usage guidance, defect, outage, or account/configuration.

4. Classify
   - Assign type, severity, priority, owner area, and confidence.
   - Use configured labels if `setup-matt-pocock-skills` has created them; otherwise use the repo's existing label vocabulary.

5. Route
   - Produce the next action: close, needs-info, convert to issue, split, merge duplicate, escalate, or schedule.
   - If using GitHub/GitLab, inspect existing labels and avoid inventing new remote labels unless asked.

6. Record evidence
   - Link commands, logs, screenshots, traces, related issues, and decisions.
   - Leave the issue body/comment in a state another engineer can act on.

## Agent Roles Without Subagents

Emulate roles sequentially unless the user explicitly asks for multi-agent work:

- Reporter advocate: preserves user pain and context.
- Reproducer: checks whether the report is observable.
- Maintainer: maps to code ownership and project priorities.
- Release manager: assesses risk, severity, and timing.
